# ServiceNow Integration

The AI agent creates ServiceNow incidents for every event it processes —
escalations stay open for human follow-up, successful remediations are
recorded as resolved tickets for ITSM audit trail.

The MCP ServiceNow server supports two modes: a local in-memory mock for
development, and a real ServiceNow Personal Developer Instance (PDI) for
staging and demos.

## 1. How the Agent Uses ServiceNow

The `decide` node picks one of three paths:

| Confidence | Path | ServiceNow Result |
|---|---|---|
| High (known failure), success | `remediate` → `servicenow_close` | Resolved ticket (opt-in via `SERVICENOW_CREATE_RESOLVED`) |
| High (known failure), retry exhausted | `remediate` → `decide` → `escalate` | Open ticket (human follow-up) |
| High (novel failure), success | `lightspeed` → `servicenow_close` | Resolved ticket (opt-in via `SERVICENOW_CREATE_RESOLVED`) |
| High (novel failure), failure | `lightspeed` → `escalate` | Open ticket (human follow-up) |
| Low | `escalate` | Open ticket (human follow-up) |

```
normalize → rag_retrieval → analyze → decide
                                        ├─ remediate ─success─→ servicenow_close → notify → audit → END
                                        │             └retry──→ decide (loop)
                                        ├─ lightspeed ─success→ servicenow_close → notify → audit → END
                                        │              └fail──→ escalate ────────→ notify → audit → END
                                        └─ escalate ────────────────────────────→ notify → audit → END
```

**Key rules:**
- The `servicenow_close` node only creates a resolved ticket when remediation actually succeeded **and** `SERVICENOW_CREATE_RESOLVED=true`. When disabled (the default), the node is a no-op — the Kafka audit topic still records all events. It never blocks the pipeline — ServiceNow errors are logged and swallowed.
- When lightspeed fails (ALS API error, AAP launch failure), the agent automatically escalates to create an open ServiceNow ticket for human follow-up, regardless of the `SERVICENOW_CREATE_RESOLVED` flag.

### Ticket Examples

**Escalation** (stays open for humans):
```
INC0010015  [AI-NOC] CrashLoopBackOff – nginx-abc in prod (edge-1)
State: New | Priority: 2 - High
```

**Auto-remediation** (created already resolved):
```
INC0010016  [AI-NOC][Resolved] OOMKilled – nginx-abc in prod (edge-1)
State: Resolved | Close: "Auto-remediated: restart-nginx via mcp-noc-aap (job 42, 8.5s)"
```

## 2. Getting a PDI

1. Go to [developer.servicenow.com](https://developer.servicenow.com) and sign up (free)
2. Request a PDI — you'll get a URL like `https://dev427607.service-now.com`
3. Note the admin password from the PDI dashboard

> **PDI hibernation:** Free PDIs sleep after inactivity. Wake them by visiting the URL in a browser, or run `make servicenow-wake`.

> **Cold-start timeout:** A freshly woken PDI may time out on incident writes for 5–10 minutes while business rules compile. Fix: open the incident form in the browser once (`/nav_to.do?uri=incident.do`, submit any incident). This warms the caches and subsequent API calls complete in ~10s.

## 3. Bootstrap the PDI

### Why Bootstrap?

A fresh PDI is a blank slate — it has no users, groups, or API policies for our agent. The bootstrap scripts automate all the ServiceNow admin work that would otherwise require clicking through dozens of UI forms. Without bootstrapping, the MCP server would get 401/403 errors when trying to create incidents.

### What It Does

The bootstrap runs 4 steps in sequence against the ServiceNow Table API using admin credentials:

```
Step 1: Create NOC Agent User
  └─ Creates a machine user `noc_agent` in sys_user
  └─ Assigns `itil` role (incident read/write) and `rest_service` role (API access)
  └─ Generates a random password, saves to local credentials file

Step 2: Configure API Access
  └─ Creates a REST API key (http_key_auth record)
  └─ Creates a Basic Auth profile (std_http_auth record)
  └─ Creates a Table API access policy (sys_api_access_policy)
  └─ Links the auth profile to the access policy (sys_auth_profile_mapping)
  └─ This allows the noc_agent user to call /api/now/table/* endpoints

Step 3: Create Test Data
  └─ Creates assignment groups: NOC-Team, Edge-Ops
  └─ Creates a sample incident assigned to NOC-Team (for validation)

Step 4: Validate
  └─ Runs the 5-test CRUD suite to confirm everything works
  └─ Cleans up the sample incident after validation
```

Each step is idempotent — running bootstrap twice won't create duplicate users or groups.

### How to Run

Set credentials in `.env` (quote values with special characters):

```bash
SERVICENOW_INSTANCE_URL="https://devXXXXX.service-now.com"
SERVICENOW_MODE=real
SERVICENOW_USERNAME=admin
SERVICENOW_PASSWORD="your-password"
```

Then run:

```bash
make servicenow-bootstrap          # runs all 4 steps
make servicenow-bootstrap-validate  # runs only step 4 (re-check connectivity)
```

You can also skip individual steps:

```bash
# Already created the user manually? Skip step 1:
cd scripts/servicenow-bootstrap && uv run python -m servicenow_bootstrap.orchestrator \
  --config config.json --skip-user --no-confirm
```

### Validation Tests

The validate command runs 5 checks against the Table API:

1. **CREATE** — POST a test incident to `/api/now/table/incident`
2. **READ** — GET it back by `sysparm_query=number=INCxxxxxxx`
3. **UPDATE** — PATCH with work notes, move state to In Progress
4. **RESOLVE** — PATCH state to Resolved with `close_code` and `resolution_code`
5. **CALLER** — Look up `NOC Agent` in `sys_user` table

All 5 must pass before deploying. The test incident is deleted after validation.

### Bootstrap Files

| File | Role |
|---|---|
| `orchestrator.py` | Runs all steps in sequence, handles config and confirmation |
| `servicenow_client.py` | Base HTTP client with session, auth, and user lookup |
| `create_noc_agent_user.py` | Step 1: user creation and role assignment |
| `create_noc_agent_api_key.py` | Step 2: API key, auth profile, access policy |
| `create_incident_test_data.py` | Step 3: assignment groups and sample incident |
| `setup_validations.py` | Step 4: CRUD tests |
| `config.json` | User details, API key name, group names |

## 4. Postman Collection

Two files in `scripts/postman/`:

| File | Purpose |
|---|---|
| `servicenow-table-api.postman_collection.json` | 8 requests in 3 folders |
| `servicenow-pdi.postman_environment.json` | Environment variables |

### Setup

1. Open Postman → **File > Import** → select both JSON files
2. Select the **ServiceNow PDI — AI Network Remediation** environment (top-right dropdown)
3. Click the eye icon → Edit → fill in `snow_instance_url`, `snow_username`, `snow_password`
4. Run **Auth > Whoami** to verify credentials

### Request Folders

| Folder | Requests |
|---|---|
| **Auth** | Whoami (verify who you're logged in as) |
| **Incidents** | Create, Get by Number, Update (Work Notes), Resolve, List |
| **Users & Groups** | Lookup NOC Agent User, Lookup NOC-Team Group |

The **Create Incident** request auto-saves `sys_id` and `ticket_number` as collection variables, so subsequent requests (Update, Resolve) use them automatically.

All requests have `disableCookies: true` to prevent ServiceNow session fallback issues.

## 5. Deploy to a Cluster

### Prerequisites

- Bootstrap completed and validated (Step 3)
- Images built and pushed (only needed if you modified `mcp-servicenow` or `agent-service`)

### Build Images (if code changed)

```bash
# MCP ServiceNow server
podman build --no-cache -t quay.io/<your-registry>/noc-mcp-servicenow:<tag> \
  --platform=linux/amd64 \
  --build-arg SERVICE_NAME=mcp-servicenow \
  --build-arg MODULE_NAME=mcp_servicenow \
  -f hub/mcp-servers/Containerfile hub/mcp-servers

# Agent service
podman build --no-cache -t quay.io/<your-registry>/noc-agent-service:<tag> \
  --platform=linux/amd64 \
  -f hub/agent-service/Containerfile hub/agent-service

# Push both
podman push quay.io/<your-registry>/noc-mcp-servicenow:<tag>
podman push quay.io/<your-registry>/noc-agent-service:<tag>
```

> **Build cache issue:** If builds fail with `idna` / `uv archive` errors, run `podman system reset -f` to clear corrupted cache volumes, then `podman login quay.io` and rebuild.

### Create the Secret

```bash
oc create secret generic noc-servicenow-secret -n <namespace> \
  --from-literal=SERVICENOW_API_KEY="" \
  --from-literal=SERVICENOW_USERNAME="admin" \
  --from-literal=SERVICENOW_PASSWORD='<your-password>'
```

### Helm Upgrade

For an existing deployment, upgrade with these overrides:

```bash
helm upgrade hub hub/helm \
  --namespace <namespace> \
  --reuse-values \
  --set servicenowMock.enabled=false \
  --set mcp-servers.mcp-servers.noc-servicenow.image.repository=quay.io/<registry>/noc-mcp-servicenow \
  --set mcp-servers.mcp-servers.noc-servicenow.image.tag=<tag> \
  --set mcp-servers.mcp-servers.noc-servicenow.env.SERVICENOW_URL=https://devXXXXX.service-now.com \
  --set mcp-servers.mcp-servers.noc-servicenow.env.SERVICENOW_MODE=real \
  --set-string agentService.servicenowInstanceUrl=https://devXXXXX.service-now.com \
  --wait --timeout 5m
```

For a fresh install via `make`:

```bash
make helm-install \
  ENABLE_SERVICENOW_MOCK=false \
  SERVICENOW_INSTANCE_URL=https://devXXXXX.service-now.com \
  HELM_EXTRA_ARGS="--set mcp-servers.mcp-servers.noc-servicenow.env.SERVICENOW_URL=https://devXXXXX.service-now.com \
                   --set mcp-servers.mcp-servers.noc-servicenow.env.SERVICENOW_MODE=real"
```

### Verify

```bash
# Check pods
oc get pods -l app=mcp-noc-servicenow   # should be Running
oc get pods -l app=servicenow-mock      # should be gone

# Trigger a demo
curl -sk -X POST https://<frontend-route>/api/demo/trigger \
  -H "Content-Type: application/json" \
  -d '{"scenario":"escalation","site":"edge-01"}'

# Check agent logs for ticket creation
oc logs deployment/hub-agent-service --tail=30 | grep -i "servicenow\|INC"

# Verify on PDI
curl -sk "https://devXXXXX.service-now.com/api/now/table/incident?sysparm_query=short_descriptionLIKEAI-NOC&sysparm_limit=5&sysparm_fields=number,short_description,state" \
  -u admin:<password> -H "Accept: application/json"
```

## 6. MCP Server — Mock vs. Real Mode

The MCP ServiceNow server (`mcp-servicenow`) works in two modes with no code changes:

| | Mock Mode | Real Mode |
|---|---|---|
| `SERVICENOW_MODE` | `mock` | `real` |
| `SERVICENOW_URL` | `http://servicenow-mock:8080` | `https://devXXXXX.service-now.com` |
| Auth | `X-API-Key` header | Basic Auth (username/password) |
| Payload format | `{"record": {...}}` | Flat JSON (ServiceNow native) |
| Incident lookup | Ticket number in URL path | `sysparm_query=number=...` then `sys_id` |

The server exposes 4 tools via LlamaStack:

| Tool | Used By | Description |
|---|---|---|
| `create_incident` | `escalate` node, `servicenow_close` node | Create a new incident |
| `resolve_incident` | `servicenow_close` node | Set state=Resolved with close code |
| `update_incident` | (future use) | Add work notes, change state |
| `get_incident` | (future use) | Fetch incident details |

### Environment Variables

| Variable | Required | Default | Description |
|---|---|---|---|
| `SERVICENOW_URL` | Yes | mock URL | Instance base URL |
| `SERVICENOW_MODE` | No | `auto` | `mock`, `real`, or `auto` (detects from URL) |
| `SERVICENOW_USERNAME` | For real | — | Basic auth username |
| `SERVICENOW_PASSWORD` | For real | — | Basic auth password |
| `SERVICENOW_API_KEY` | For mock | — | API key for mock service |
| `SERVICENOW_CALLER_NAME` | No | `NOC Agent` | Caller field on new incidents |

### Agent Service Environment Variables

| Variable | Default | Description |
|---|---|---|
| `SERVICENOW_INSTANCE_URL` | `""` | PDI base URL (used for Slack ticket links) |
| `SERVICENOW_CREATE_RESOLVED` | `false` | Create pre-resolved tickets for successful auto-remediations. When `false`, only escalations produce ServiceNow tickets; the Kafka audit topic still records all events. Set to `true` for full ITSM visibility. |

## 7. Troubleshooting

| Problem | Fix |
|---|---|
| PDI returns "maximum execution time exceeded" | Open `/nav_to.do?uri=incident.do` in browser, create one incident manually to warm caches |
| 401 "User is not authenticated" | Check `noc-servicenow-secret` has correct credentials, restart MCP pod: `oc rollout restart deployment/mcp-noc-servicenow` |
| 403 "Resolution code mandatory" on resolve | The `resolution_code` field is required on some PDIs — already handled in our code (sends both `close_code` and `resolution_code`) |
| `bash: Agent: command not found` when sourcing `.env` | Quote values with spaces or special chars: `SERVICENOW_CALLER_NAME="NOC Agent"` |
| Build fails with `idna` / `uv archive` error | Run `podman system reset -f`, re-login to quay.io, rebuild |
| Helm upgrade fails with nil pointer on slack/lightspeed | Add `--set agentService.slack.enabled=false --set-string agentService.lightspeed.tokenSecretName=""` (and other defaults) when using `--reuse-values` |
| PDI hibernating | Visit the URL in a browser or run `make servicenow-wake` |
